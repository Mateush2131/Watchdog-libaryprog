# Техническое задание
## Проект: "Автоматический архиватор лабораторных работ"

### 1. Общая информация
*   **Название проекта:** Тз по библиотеке wathcdog и автоматический архиватор работ ).
*   **Заказчик:** Учебное заведение TOP академия .
*   **Исполнитель:** Студент Копычев Матвей.
*   **Цель проекта:** Разработать программу, которая в реальном времени отслеживает изменения в рабочей директории студента и автоматически создаёт резервные копии файлов лабораторных работ, предотвращая их потерю.

### 2. Описание задачи (что нужно сделать)
Необходимо разработать программу-демон (фоновую службу), которая будет незаметно работать на компьютере студента. Её задача — следить за указанной папкой (например, "Лабораторные") и при любом сохранении файла с лабораторной работой (например, `.py`, `.ipynb`, `.txt`) мгновенно создавать его датированную резервную копию в отдельной папке. Это решает проблему случайного удаления, перезаписи или повреждения файлов в процессе работы.

### 3. Функциональные требования (что система должна уметь)
Система должна:
1.  Мониторить файловую систему в реальном времени, используя библиотеку `watchdog`.
2.  Автоматически обнаруживать события создания (`created`) и изменения (`modified`) файлов в целевой директории.
3.  Фильтровать события, обрабатывая только файлы с определёнными расширениями: `.py`, `.ipynb`, `.txt`, `.md`.
4.  Игнорировать события, связанные с директориями (папками) и временными/служебными файлами (например, файлы, начинающиеся с `~` или `.tmp`).
5.  При срабатывании события создавать резервную копию файла.
6.  Сохранять резервную копию в отдельную директорию `backup` с уникальным именем, включающим исходное имя файла и отметку времени (дата_время).
7.  Вести простой текстовый лог (`log.txt`), в который записываются все операции (какой файл и когда был скопирован).
8.  Корректно запускаться и останавливаться по команде пользователя (например, по нажатию `Ctrl+C` в терминале).

### 4. Нефункциональные требования (ограничения)
*   **Технологический стек:**
    *   Язык программирования: **Python 3.8+**.
    *   Ключевая библиотека: **`watchdog`**.
    *   Формат хранения данных: **Файловая система** (папки и файлы).
    *   Интерфейс: **Консольное приложение (CLI)**.
*   **Платформа:** Программа должна запускаться и корректно работать на операционных системах **Windows 10/11** и **Linux** (Ubuntu/Debian).
*   **Производительность:** Программа не должна создавать заметную нагрузку на систему при отслеживании директории с количеством файлов до 1000.
*   **Надёжность:** Программа должна корректно обрабатывать ошибки (например, отсутствие прав на запись в папку `backup`) без аварийного завершения, записывая информацию об ошибке в лог.

### 5. Требования к интерфейсу
Интерфейс — консольный (командная строка).
1.  Программа запускается из терминала командой: `python backup_watcher.py [путь_к_папке_для_слежки]`.
2.  Если путь не указан, по умолчанию слежка ведётся за текущей директорией, откуда запущена программа.
3.  При запуске выводится информационное сообщение: `[INFO] Наблюдение начато за директорией: <указанный_путь>. Для остановки нажмите Ctrl+C.`
4.  В процессе работы в консоль в реальном времени выводятся ключевые события: `[СОБЫТИЕ] Файл "<имя_файла>" скопирован в архив.`
5.  При получении сигнала остановки (`Ctrl+C`) программа выводит сообщение `[INFO] Остановка наблюдателя...` и завершает работу.

### 6. Входные и выходные данные
*   **Входные данные:**
    1.  Путь к директории для наблюдения (аргумент командной строки, опционально).
    2.  Поток событий файловой системы (создание, изменение файлов).
*   **Выходные данные:**
    1.  Резервные копии файлов в директории `./backup/` (или `[путь_наблюдения]/backup/`).
    2.  Файл лога `log.txt` в корневой директории программы.
    3.  Информационные и error-сообщения в консоли.

### 7. Критерии приёмки 
Работа считается выполненной и принимается, если:
1.  Реализованы **все функциональные требования** из раздела 3.
2.  Программа успешно запускается командой из п.5.1 и стабильно работает в фоновом режиме.
3.  При создании или изменении текстового (`*.txt`) или Python-файла (`*.py`) в наблюдаемой директории его резервная копия с timestamp в имени появляется в папке `backup` в течение 1-2 секунд.
4.  В консоли отображаются понятные сообщения о событиях.
5.  Записи о каждом событии копирования добавляются в файл `log.txt`.
6.  Программа корректно останавливается по `Ctrl+C`, освобождая все ресурсы.
7.  Весь исходный код проекта, включая файл `requirements.txt`, размещён в **GitHub-репозитории**.
8.  В корне репозитория присутствует рабочий файл `README.md` с описанием проекта и инструкцией по запуску.
